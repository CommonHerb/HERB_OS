# Hierarchical containment formalisms and what HERB can learn from them

**No single existing formalism does exactly what HERB is about to do — but four independent traditions each solved a piece of the puzzle, and their successes and failures map directly onto HERB's design decisions.** HERB's specific combination of typed containers, entity identity, MOVE-only primitives, and operation-set-as-constraint with hierarchical scoping is genuinely novel. The closest existing match is Lomazova's Nested Petri Nets with value semantics, where van Hee and Lomazova formally proved that careful selection of basic operations enforces model correctness by construction — the exact principle HERB embodies. However, P systems' symport/antiport rules capture the "pure movement" philosophy, Boxed Ambients capture permanent boundaries without destructive `open`, and Zircon's kernel model independently validates MOVE-as-default for cross-boundary resource transfer. HERB is charting new territory in its *combination*, but each individual idea has deep prior art with known pitfalls worth studying.

## The four formalisms share a hidden structure

Each formalism addresses hierarchical containment differently, but a common architecture emerges: **boundaries create isolation, rules define what crosses boundaries, and the rule set determines reachable states**. The differences lie in what crosses boundaries, how crossing is authorized, and whether boundaries are permanent.

**Bigraphs** (Milner, 2001–2009) decompose systems into two orthogonal structures: a *place graph* (tree of nested containment) and a *link graph* (hypergraph of connections). The place graph is HERB's container hierarchy. The link graph has no direct HERB analogue — it allows connections to freely cross all containment boundaries, which is precisely the permissiveness HERB rejects. Reaction rules can arbitrarily restructure both graphs simultaneously, making bigraphs maximally general. A MOVE from container A to container B is expressible as a bigraphical reaction rule: `Container_A.(Entity | site₀) | Container_B.site₁ → Container_A.site₀ | Container_B.(Entity | site₁)`. But bigraphs permit far more — rules can create nodes, destroy links, rearrange entire hierarchies in a single step. **Isolation is not automatic in bigraphs.** Two sibling nodes can freely share hyperedges regardless of nesting depth. To get isolation, you must layer on additional machinery: passive controls (which block reactions inside a node), binding bigraphs (which scope names), and sorting disciplines. HERB achieves by *absence of operations* what bigraphs achieve through accumulated restrictions.

**Ambient calculus** (Cardelli & Gordon, 1998) makes the boundary itself the central concept. An ambient `n[P]` is a named boundary containing process P. Three capabilities — `in n` (enter), `out n` (exit), `open n` (dissolve) — govern all cross-boundary movement. The crucial difference from HERB: **movement in the ambient calculus moves an entire ambient with all its contents**, not individual entities. When ambient `n` enters ambient `m`, everything nested inside `n` comes along. Communication is strictly local — processes can only exchange messages within the same ambient. Cross-ambient communication requires physically moving a messenger ambient into the target. The `open` capability, which **permanently destroys a boundary**, is the calculus's most criticized feature and motivated two major successor formalisms. **Boxed Ambients** (Bugliesi, Castagna, Crafa, 2001) dropped `open` entirely and added parent-child communication primitives — making boundaries permanent while allowing data exchange across exactly one level of hierarchy. This is architecturally very close to HERB's model. **Safe Ambients** (Levi & Sangiorgi, 2000) kept `open` but added co-capabilities, requiring both sides to agree before any boundary crossing.

**P systems** (Păun, 1998) provide the purest expression of HERB's core principle. In symport/antiport P systems, **objects are never created, destroyed, or transformed — they only change location**. A symport rule `(AB, in)` moves objects A and B together into a membrane; an antiport rule `(B, in; C, out)` moves B in while C moves out simultaneously. The set of available rules defines exactly which redistributions of objects across membranes are reachable. If no rule moves object `a` out of membrane 2, then `a` is permanently trapped there. This is precisely "the operation set IS the constraint system." Transfer works only between adjacent compartments (parent↔child), enforcing strict locality. The main limitations for HERB's purposes: objects are untyped symbols in multisets with no individual identity (two copies of `a` are indistinguishable), the maximal parallelism assumption is biologically motivated but impractical for programming, and nondeterminism is fundamental to most universality results.

**Nested Petri Nets** (Lomazova, 2000) come closest to HERB's planned architecture. In NP-nets, tokens in a system net can themselves be marked Petri nets ("net-tokens"), each with their own private internal places and markings. This directly models HERB's goal: an entity (process) that owns its own containers (internal places with their own tokens). Four firing modes govern interaction: autonomous system steps move net-tokens between system places without affecting their internals; autonomous element steps let a net-token evolve internally without affecting the system; **vertical synchronization** fires transitions in the system net and net-tokens simultaneously (the key cross-boundary mechanism); and **horizontal synchronization** lets net-tokens in the same place interact. Under value semantics, each net-token is a fully independent copy — no aliasing, no shared state, complete isolation by construction. Van Hee and Lomazova (ICATPN 2006) proved that **by careful selection of basic operations on nets, one can obtain a modeling formalism that enforces correctness of models** — the theoretical validation of HERB's operation-set-as-constraint principle.

## How each formalism handles isolation and cross-boundary operations

The table below distills the critical architectural differences:

| Dimension | Bigraphs | Ambient Calculus | P Systems | Nested Petri Nets |
|---|---|---|---|---|
| **Hierarchy** | Place graph (forest) | Ambient nesting (tree) | Membrane tree | System net + net-tokens |
| **Isolation default** | None — links cross all boundaries | Strong — local communication only | Strong — adjacent transfer only | Strong — private markings per net-token |
| **Cross-boundary ops** | Reaction rules (arbitrary restructuring) | in/out/open capabilities | Symport/antiport rules | Vertical/horizontal synchronization |
| **Boundary persistence** | Controls can be active/passive/atomic | `open` destroys boundaries | Dissolution (δ) destroys membranes | Net-tokens persist unless consumed |
| **Entity identity** | Nodes have identity | Ambients have names | Objects lack identity (multisets) | Net-tokens have identity (value) or reference |
| **Operation constraint** | Any rule definable; must add sorting | Capabilities gate movement; types added post-hoc | Rules define all reachable states | Operations enforce correctness by construction |

**What each formalism got right that HERB should learn from:**

Bigraphs' key insight is that containment and connectivity are orthogonal concerns. If HERB entities ever need references or connections to entities in other scopes, those references form a link graph whether HERB names it or not. **HERB should decide explicitly whether cross-scope references exist** — and if they do, whether they can cross container boundaries freely (bigraph-style) or are confined within scopes. The bigraph community also discovered that matching (finding where a rule applies in a complex nested structure) is NP-complete for combined place+link graphs. If HERB's Tensions system needs pattern matching over hierarchical container states, this complexity bound applies.

The ambient calculus teaches through its **failures**. The `open` capability — which destroys boundaries — was a mistake that took two major research programs to fix. HERB's decision to make containers permanent is validated by the entire Boxed Ambients line of work. The ambient calculus also revealed that even the minimal fragment with just `in` and `out` (no `open`, no communication) is Turing-complete. If HERB's scoped containers allow unbounded nesting depth with recursive entity creation, static analysis of container structure will be undecidable. The ambient calculus's "grave interference" problem — where an entity can enter another entity's scope without consent — motivated Safe Ambients' co-capabilities. **HERB should consider whether MOVE requires authorization from the destination container's owner**, not just the entity performing the move.

P systems validate that pure-movement computation is both expressively complete and analytically tractable in restricted cases. Symport/antiport systems, where objects only move and never transform, are Turing-complete with remarkably small configurations (one membrane, antiport weight 2). But the P systems community also found that **deterministic pure-movement systems are significantly weaker than nondeterministic ones** — strongly deterministic systems without control mechanisms accept only sub-regular sets. If HERB's MOVE is deterministic (specifying exact source and target), this constrains expressive power relative to nondeterministic alternatives. P systems' strict one-level-at-a-time transfer is also significant: if HERB allows MOVE between arbitrary containers regardless of hierarchy depth, it has a more flexible (but harder to reason about) communication model.

Nested Petri Nets provide the most directly applicable formalization. NP-nets' **four-mode firing** maps cleanly onto HERB operations: autonomous system steps = moving entities between containers at the outer level; autonomous element steps = entities evolving their own internal state; vertical synchronization = cross-boundary operations where a container-level action and an entity-level action must agree; horizontal synchronization = IPC between entities in the same scope. The key limitation: NP-nets synchronize on control flow (transition labels) but **cannot describe data sharing between levels**. Reference Nets (Kummer, 2002) solved this by adding synchronous channels with parameter passing, at the cost of Turing-completeness and loss of decidability. HERB will face the same trade-off: pure synchronization labels enable compositional verification; data-carrying channels enable practical programming.

## Real kernels validate the model from engineering practice

Three production OS kernels independently converged on a model that mirrors HERB's architecture: **each process owns a per-process namespace of typed, rights-bearing references to kernel-managed objects, with controlled transfer of references between namespaces**.

Linux's `task_struct` is a bundle of pointers to independently sharable resource containers — `files_struct` (FD table), `mm_struct` (memory map), `signal_struct`, `nsproxy` (namespace memberships), `cred` (credentials/capabilities). The **CLONE_* flags** make each dimension independently toggleable between SHARE and ISOLATE: `CLONE_FILES` shares the FD table; its absence copies it. This creates a continuum from full sharing (threads: `CLONE_VM | CLONE_FILES | CLONE_SIGHAND | ...`) to full isolation (containers: `CLONE_NEWPID | CLONE_NEWNET | CLONE_NEWNS | ...`). Linux's 8 namespace types each virtualize a different global resource — PID namespace isolates process ID space, network namespace isolates the entire network stack, mount namespace isolates filesystem views. FD passing via `SCM_RIGHTS` is a **copy** operation: both sender and receiver retain access to the underlying kernel file description.

seL4's capability model is the most formally rigorous. Each thread's CSpace (capability space) is a directed graph of CNodes containing typed capabilities — unforgeable tokens encoding {object reference + type + rights}. **There are no ambient authorities** — no global filesystem, no process table, no implicit access. A thread can only touch objects it holds capabilities for. The capability derivation tree (CDT) tracks parent-child relationships: `Retype` creates typed objects from untyped memory; `Mint` creates derived capabilities with reduced rights; `Revoke` removes all capabilities derived from a given capability across all CSpaces system-wide. seL4's formal verification (Isabelle/HOL, ~500K lines of proof) guarantees integrity, confidentiality, and information-flow noninterference — mathematically proving that the capability system provides unbreachable isolation when configured with separate CSpaces.

**Zircon (Fuchsia) provides the strongest validation of HERB's MOVE primitive.** Handles are process-local references to kernel objects carrying a rights bitmask. When a handle is written into a channel, **it is atomically removed from the sender's handle table** — this is a true MOVE, not a copy. The sender loses access unless they explicitly `zx_handle_duplicate()` first. New processes start with **zero handles** and receive all authority through the bootstrap channel. The job tree creates hierarchical containment where policies can only be made more restrictive downward. Rights attenuation (never amplification) mirrors capability theory's monotonicity. Zircon independently arrived at MOVE-as-default because it simplifies reasoning about who has access: at most one process holds a given handle unless duplication was explicit.

The cross-cutting pattern is clear. What **must** be isolated per-process includes reference tables (FD/capability/handle tables), address spaces, execution state, notification/signal state, and identity/credentials. Controlled sharing follows a universal pattern: kernel-mediated transfer of typed references between per-process namespaces, with rights that can only be attenuated during transfer. The key architectural choice is **MOVE vs. COPY as default**: Linux chose copy (SCM_RIGHTS); Zircon chose move (channel write removes handle); seL4 supports both. HERB's MOVE-only approach aligns with Zircon's philosophy and is strictly more conservative than Linux's.

## Where HERB stands relative to prior art

HERB's scoped containers correspond to the **place graph half** of bigraphs — a forest of typed containment nodes. HERB's MOVE corresponds to a specific, restricted class of bigraphical reaction rules that modify only parent assignments while preserving everything else. HERB's typed containers correspond to bigraphical controls with sorting constraints. But HERB deliberately lacks (and benefits from lacking) bigraphs' link graph — the ability for arbitrary cross-scope connections. If HERB later adds any form of inter-entity reference or channel, it will effectively introduce a link graph, and the full bigraphical structure will emerge.

The formalism closest to HERB's complete design is **a fusion of NP-nets (for hierarchy and isolation) with P systems' symport/antiport philosophy (for pure-movement semantics)**. In this synthesis: containers are system-net places typed by element nets; entities are net-tokens with private internal markings; MOVE is a system-net transition that transfers a net-token between places; Tensions map to synchronization labels or channel patterns that trigger coordinated firing; and the operation set defines exactly which net-token transfers are possible, making invalid states structurally unreachable.

The specific differences that make HERB's combination novel are threefold. First, **entities have both identity and typed internal state** — P systems' objects lack identity; bigraphs' nodes have identity but no "internal marking"; NP-nets come closest but their net-tokens evolve via Petri net rules rather than typed container operations. Second, **MOVE is the sole primitive, not one of several** — all formalisms support additional operations (bigraphs allow arbitrary reaction rules; ambients have in/out/open; P systems have evolution rules alongside symport/antiport; NP-nets have four firing modes). HERB's restriction to MOVE-only is more radical. Third, **Tensions (declarative reactive patterns) have no direct analogue** — NP-nets' synchronization labels and Reference Nets' channels are the closest, but Tensions appear to be declarative constraints that trigger MOVEs rather than imperative synchronization requests.

## Known pitfalls HERB should anticipate

The formalisms reveal five recurring failure modes. **Boundary destruction is always a mistake** — the ambient calculus's `open`, P systems' membrane dissolution (δ), and any mechanism that collapses scoping boundaries creates irreversible loss of isolation and security headaches. HERB's permanent containers are correct. **Unrestricted nesting depth plus recursion yields undecidability** — if entities can create entities that create entities to arbitrary depth, determining whether a particular container configuration is reachable becomes undecidable (proven for ambient calculus with just in/out). HERB should consider bounding nesting depth or restricting recursive entity creation if static analysis is desired. **Cross-scope references undermine isolation** — bigraphs' orthogonality between place and link graphs means links cross all boundaries freely, which is elegant mathematically but forces you to re-engineer isolation from scratch. If HERB introduces any form of inter-scope reference, it should be confined to specific, typed channels rather than free-form linking. **The matching problem is expensive** — finding where a pattern (Tension) applies in a hierarchical container state involves sub-tree or sub-graph isomorphism, which is NP-complete in the general case (proven for bigraphs). HERB's type system on containers may constrain this enough to remain tractable, but it's worth analyzing. **Move-vs-copy semantics must be chosen deliberately** — the kernel world shows this is an architectural decision with deep consequences. Zircon's move-by-default simplifies ownership reasoning; Linux's copy-by-default simplifies programming but complicates access tracking. HERB's MOVE-only choice is the strictest option and matches Zircon's philosophy, but should provide an explicit COPY-then-MOVE pattern (like Zircon's `handle_duplicate` before channel write) when shared access is intentionally needed.

## Conclusion

HERB is not reinventing any single existing formalism, but it is reinventing a specific *combination* of ideas that each appear independently in prior art. The place-graph half of bigraphs provides the mathematical foundation for container hierarchy. Symport/antiport P systems validate that pure-movement computation is expressively complete. Boxed Ambients prove that permanent boundaries with structured cross-boundary communication are superior to destructive `open`. Nested Petri Nets with value semantics provide the most complete formal model for entities-owning-containers with isolation guarantees, and van Hee & Lomazova's proof that "operations enforce correctness by construction" is the theoretical anchor for HERB's constraint philosophy. Zircon's kernel design validates MOVE-as-default from engineering practice. The novel contribution HERB makes is unifying these into a single coherent system where typed containers + entity identity + MOVE-only + declarative Tensions + operation-set-as-constraint all reinforce each other. The prior art suggests this combination is sound — every individual piece has been validated — but the synthesis is genuinely new territory. The most productive next step would be to formalize HERB's scoped containers as a restricted class of NP-net with symport/antiport-style movement rules, which would let HERB inherit the existing decidability and compositionality results while maintaining its distinctive character.
