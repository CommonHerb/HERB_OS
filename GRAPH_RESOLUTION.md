# HERB v2: The Resolution Algorithm

How execution actually works. This is where the graph comes alive.

---

## THE FUNDAMENTAL QUESTION

HERB v1 had a simple execution model: each tick, fire rules until fixpoint, advance.
Clear, predictable, easy to implement.

HERB v2 is more complex. We have:
- **Deltas** trying to happen
- **Constraints** that must hold
- **Goals** that want to become true
- **Multiple strategies** for different subproblems

How do these interact? What's the order of operations? When does execution "stop"?

---

## INSIGHT: EXECUTION IS REACTION TO DISTURBANCE

The graph is at rest when:
1. No pending deltas
2. All constraints satisfied
3. All goals either achieved, failed, or suspended

Execution is the process of returning to rest after a disturbance.

A disturbance is anything that creates pending work:
- External input (player presses button, packet arrives, timer fires)
- Goal activation (NPC decides to go somewhere)
- Constraint violation (something became invalid, needs repair)

The resolution algorithm takes a disturbed graph and brings it back to rest.

---

## THE RESOLUTION LOOP

```
fn resolve(graph: &mut Graph) -> ResolutionResult {
    let mut iterations = 0;
    let max_iterations = 10000;  // safety limit

    loop {
        iterations += 1;
        if iterations > max_iterations {
            return ResolutionResult::Diverged;
        }

        // Phase 1: Apply pending deltas
        let delta_result = apply_pending_deltas(graph);
        if delta_result.has_violations() {
            return ResolutionResult::ConstraintViolation(delta_result.violations);
        }

        // Phase 2: Propagate constraint consequences
        let propagation_result = propagate_constraints(graph);
        // This may add new pending deltas

        // Phase 3: Pursue active goals
        let goal_result = pursue_goals(graph);
        // This may add new pending deltas

        // Phase 4: Check for quiescence
        if graph.pending_deltas.is_empty() &&
           propagation_result.no_new_work() &&
           goal_result.no_new_work()
        {
            return ResolutionResult::Quiescent { iterations };
        }
    }
}
```

This is the outer loop. Let's look at each phase.

---

## PHASE 1: APPLY PENDING DELTAS

Deltas are queued, not applied immediately. This allows:
- Batching (multiple deltas in one "moment")
- Ordering (causal dependencies respected)
- Validation (check before committing)

```
struct PendingDelta {
    delta: Delta,
    priority: i32,              // higher = applied first
    causal_deps: Vec<u64>,      // must be applied after these deltas
    source: DeltaSource
}

enum DeltaSource {
    External,                   // from outside the system
    Constraint(u64),            // generated by constraint maintenance
    Goal(u64),                  // generated by goal pursuit
    Rule(u64)                   // generated by a v1-style rule (for compatibility)
}
```

### Delta Application Order

Deltas are applied in order determined by:
1. **Causal dependencies**: If delta A depends on delta B, B is applied first
2. **Priority**: Among independent deltas, higher priority first
3. **Source**: External > Constraint > Goal > Rule (when priority is equal)

This is a topological sort with tie-breaking.

```
fn apply_pending_deltas(graph: &mut Graph) -> DeltaApplicationResult {
    let mut result = DeltaApplicationResult::new();

    // Sort by causal order + priority
    let ordered = topological_sort(&graph.pending_deltas);

    for pending in ordered {
        // Check preconditions
        if let Err(violation) = check_preconditions(graph, &pending.delta) {
            result.add_violation(pending, violation);
            continue;  // skip this delta, try others
        }

        // Apply the delta
        apply_single_delta(graph, &pending.delta);

        // Check postconditions
        if let Err(violation) = check_postconditions(graph, &pending.delta) {
            // Rollback? Or record violation and continue?
            // Design choice: record and continue, let constraint propagation fix it
            result.add_violation(pending, violation);
        }

        // Check invariants affected by this delta
        let affected_invariants = get_affected_invariants(graph, &pending.delta);
        for inv_id in affected_invariants {
            if let Err(violation) = check_constraint(graph, inv_id) {
                result.add_violation(pending, violation);
            }
        }

        result.add_applied(pending.delta.id);
    }

    // Clear applied deltas from pending queue
    graph.pending_deltas.retain(|d| !result.applied.contains(&d.delta.id));

    result
}
```

### Handling Violations

When a delta violates a constraint, what happens?

**Option A: Reject the delta** — Don't apply it. The system stays valid but the delta is lost.

**Option B: Apply and repair** — Apply the delta, then let constraint propagation fix the violation.

**Option C: Transactional** — Group related deltas, apply all or none.

The right choice depends on the domain:
- Game: Usually Option A (can't buy what you can't afford)
- OS: Usually Option B (processes die, system recovers)
- Browser: Usually Option C (DOM update is atomic)

We support all three via delta annotations:

```
PendingDelta {
    ...
    on_violation: ViolationPolicy
}

enum ViolationPolicy {
    Reject,                     // don't apply if violated
    ApplyAndRepair,             // apply, let propagation fix
    Transaction(Vec<u64>)       // apply all these together or none
}
```

## PHASE 2: PROPAGATE CONSTRAINT CONSEQUENCES

Some constraints don't just validate — they DERIVE. When their inputs change,
they compute new values.

Example: `player.total_damage = sum of all damage_taken tuples for this player`

This isn't an invariant (which would reject invalid states). It's a **derived relation** — its value is computed from other relations.

### Constraint Types Revisited

```
ConstraintKind {
    // Validation constraints (reject violations)
    Invariant,
    Precondition { delta_type: u64 },
    Postcondition { delta_type: u64 },

    // Derivation constraints (compute values)
    Derived {
        target_relation: u64,
        target_roles: Vec<String>,  // which roles are computed
        computation: Expr
    },

    // Maintenance constraints (generate repair deltas)
    Maintenance {
        condition: ConstraintExpr,  // when this becomes false...
        repair: RepairStrategy      // ...do this to fix it
    }
}

enum RepairStrategy {
    AssertDelta(DeltaTemplate),     // assert this to fix
    RetractMatching(QueryPattern),  // retract tuples matching this
    Custom(fn(&Graph, &Violation) -> Vec<Delta>)  // custom repair logic
}
```

### Propagation Algorithm

```
fn propagate_constraints(graph: &mut Graph) -> PropagationResult {
    let mut result = PropagationResult::new();
    let mut dirty: HashSet<u64> = HashSet::new();  // relation types that changed

    // Mark relations touched by recent deltas as dirty
    for delta in &graph.recently_applied {
        dirty.insert(delta.tuple.type_id);
    }
    graph.recently_applied.clear();

    // Iterate until no more propagation
    loop {
        let mut new_deltas: Vec<PendingDelta> = vec![];

        // Check derived constraints whose inputs are dirty
        for (constraint_id, constraint) in &graph.constraints {
            if let ConstraintKind::Derived { target_relation, computation, .. } = &constraint.kind {
                // Does this constraint depend on any dirty relation?
                let deps = get_constraint_dependencies(constraint);
                if deps.iter().any(|d| dirty.contains(d)) {
                    // Recompute derived values
                    let new_values = evaluate_derivation(graph, computation);
                    let deltas = generate_derivation_deltas(graph, *target_relation, new_values);
                    new_deltas.extend(deltas);
                }
            }

            if let ConstraintKind::Maintenance { condition, repair } = &constraint.kind {
                // Is the maintenance condition violated?
                if !eval_constraint_expr(graph, condition, &HashMap::new()) {
                    // Generate repair deltas
                    let repair_deltas = generate_repair_deltas(graph, repair, *constraint_id);
                    new_deltas.extend(repair_deltas);
                }
            }
        }

        if new_deltas.is_empty() {
            break;
        }

        // Apply the new deltas (this may create more dirty relations)
        for pending in new_deltas {
            dirty.insert(pending.delta.tuple.type_id);
            graph.pending_deltas.push(pending);
        }

        // Apply pending deltas
        let apply_result = apply_pending_deltas(graph);
        result.merge(apply_result);
    }

    result
}
```

### This Subsumes HERB v1 Rules

A HERB v1 rule like:
```
WHEN damage(E, D) AND hp(E, H) THEN hp(E, H - D), RETRACT hp(E, H)
```

Is equivalent to a derived constraint:
```
Derived {
    target_relation: hp,
    computation: "for each entity E: initial_hp(E) - sum(damage(E, _))"
}
```

The rule-based view and the constraint-based view are equivalent. But the constraint
view is more declarative — you say WHAT hp should be, not HOW to update it.

## PHASE 3: PURSUE GOALS

Goals are desires. The system tries to make them true.

```
fn pursue_goals(graph: &mut Graph) -> GoalPursuitResult {
    let mut result = GoalPursuitResult::new();

    for goal_id in &graph.active_goals.clone() {
        let goal = &graph.goals[goal_id];

        // Already achieved?
        if eval_constraint_expr(graph, &goal.target, &HashMap::new()) {
            mark_goal_achieved(graph, *goal_id);
            result.achieved.push(*goal_id);
            continue;
        }

        // Deadline passed?
        if let Some(deadline) = goal.deadline {
            if graph.current_timestamp > deadline {
                mark_goal_failed(graph, *goal_id, "deadline passed");
                result.failed.push(*goal_id);
                continue;
            }
        }

        // Try to make progress
        let progress = try_goal_progress(graph, *goal_id);
        match progress {
            GoalProgress::Advanced(deltas) => {
                for delta in deltas {
                    graph.pending_deltas.push(PendingDelta {
                        delta,
                        priority: goal.priority,
                        causal_deps: vec![],
                        source: DeltaSource::Goal(*goal_id),
                        on_violation: ViolationPolicy::Reject
                    });
                }
                result.progressed.push(*goal_id);
            }
            GoalProgress::Blocked(reason) => {
                // Try to decompose into subgoals
                if let Some(subgoals) = try_decompose_goal(graph, *goal_id) {
                    for subgoal_id in subgoals {
                        graph.active_goals.push(subgoal_id);
                    }
                    result.decomposed.push(*goal_id);
                } else {
                    // Can't decompose, goal is stuck
                    result.stuck.push((*goal_id, reason));
                }
            }
            GoalProgress::NoAction => {
                // Goal is waiting for something external
            }
        }
    }

    result
}
```

### Goal Progress Strategies

How does `try_goal_progress` work? It depends on the goal structure.

**Strategy 1: Direct Delta**
If there's a delta type whose postcondition matches the goal, try to apply it.

Goal: `player_1 has sword_42`
→ Find delta type `purchase` with postcondition `buyer.inventory contains item`
→ Check if preconditions can be satisfied
→ If yes, generate purchase delta

**Strategy 2: Pathfinding**
If the goal is spatial (`entity at location`), use A* or similar.

Goal: `npc_7 at market`
→ Recognize this as spatial goal
→ Run A* from current location to market
→ Generate movement deltas for the path

**Strategy 3: Planning (HTN)**
If the goal is complex, decompose into subgoals.

Goal: `npc_farmer completes daily_routine`
→ Decompose: [go_to_field, harvest, go_to_market, sell]
→ Each subgoal uses its own strategy

**Strategy 4: Constraint Satisfaction**
If the goal is constraint-like, use constraint solving.

Goal: `layout is valid` (all elements fit in their containers)
→ Run constraint propagation with the goal as a soft constraint
→ Generate deltas that move toward satisfaction

The system CHOOSES the strategy based on goal annotations or goal structure:

```
Goal {
    ...
    strategy_hint: Option<StrategyHint>
}

enum StrategyHint {
    DirectDelta,
    Spatial,
    Planning,
    ConstraintSolving,
    Auto  // system figures it out
}
```

## CAUSAL ORDERING VS TICKS

HERB v1 uses ticks — discrete time steps. All rules fire within a tick, then time advances.

HERB v2 uses causal ordering — events happen in the order required by their dependencies.

But ticks are USEFUL for games. We want both.

### Reconciliation: Ticks as Batched Resolution

A "tick" in v2 is:
1. Collect all external inputs since last tick
2. Run resolution until quiescent
3. Advance logical time

```
fn tick(graph: &mut Graph, external_inputs: Vec<Delta>) {
    // Queue external inputs
    for input in external_inputs {
        graph.pending_deltas.push(PendingDelta {
            delta: input,
            priority: 100,  // external inputs are high priority
            causal_deps: vec![],
            source: DeltaSource::External,
            on_violation: ViolationPolicy::Reject
        });
    }

    // Resolve until quiescent
    let result = resolve(graph);

    match result {
        ResolutionResult::Quiescent { iterations } => {
            // Success - advance time
            graph.current_tick += 1;
            graph.current_micro = 0;
        }
        ResolutionResult::Diverged => {
            // Resolution didn't terminate - this is a bug in the constraints/goals
            panic!("Resolution diverged at tick {}", graph.current_tick);
        }
        ResolutionResult::ConstraintViolation(violations) => {
            // Some constraints couldn't be satisfied
            // Log and continue? Depends on domain.
            log_violations(&violations);
            graph.current_tick += 1;
        }
    }
}
```

For event-driven systems (OS, browser), you don't use `tick()`. You use `resolve()`
directly whenever events arrive.

## PROVENANCE THROUGH RESOLUTION

Every delta records WHY it happened:

```
Provenance {
    cause: ProvCause,
    source_deltas: Vec<u64>,    // deltas that triggered this one
    constraint_id: Option<u64>, // if from constraint propagation
    goal_id: Option<u64>,       // if from goal pursuit
    rule_id: Option<u64>,       // if from v1-style rule
    tick: u64,
    micro: u32
}
```

This gives us full traceability:
- "Why does player_1 have 90 gold?" → "Delta #2847 (purchase) subtracted 110"
- "Why did delta #2847 happen?" → "Goal #15 (acquire sword) generated it"
- "Why was goal #15 active?" → "Player input at tick 100"

The entire causal chain is navigable.

## STRATEGY SELECTION

The system needs to choose strategies for:
- Constraint propagation (forward vs backward, local vs global)
- Goal pursuit (direct, pathfinding, planning, constraint solving)
- Conflict resolution (which delta wins when they conflict?)

### Domain Annotations

Rather than trying to infer the right strategy (which is AI-complete), we use annotations:

```
RelationType {
    ...
    domain_hints: DomainHints
}

DomainHints {
    spatial: bool,          // involves positions in space
    temporal: bool,         // involves time ordering
    numeric: bool,          // involves arithmetic
    hierarchical: bool,     // tree-structured
    resource: bool          // conserved quantity
}

Constraint {
    ...
    propagation_hint: PropagationHint
}

enum PropagationHint {
    Local,                  // only affects nearby entities
    Global,                 // affects entire domain
    Monotonic,              // once true, stays true
    Functional              // single-valued (like HERB v1 FUNCTIONAL)
}
```

These hints help the resolver choose algorithms without full semantic analysis.

### Default Strategies

When no hint is given:
- **Numeric constraints**: Forward propagation with interval arithmetic
- **Spatial goals**: A* with Euclidean heuristic
- **Boolean constraints**: Unit propagation (SAT-solver style)
- **Existential goals**: Backward search for satisfying assignments
- **Aggregate constraints**: Incremental maintenance

## WHAT'S GENUINELY NEW HERE

Let me check myself against The Standard.

**Not new:**
- Constraint propagation (from constraint programming, 1970s)
- Goal pursuit (from planning systems, 1970s)
- Causal ordering (from distributed systems, 1978)
- Provenance tracking (from database lineage, 1990s)

**Arguably new:**
- Constraints, goals, and deltas in ONE unified framework with shared representation
- The graph being simultaneously program, state, and execution trace
- Strategy selection based on domain hints rather than requiring the programmer to choose
- v1-style rules as a special case of derived constraints (unifying forward-chaining and constraint propagation)

The novelty isn't in any single mechanism. It's in the UNIFICATION. Most systems have
rules OR constraints OR goals. This has all three, with the same representation,
the same provenance, the same execution framework.

Is that novel ENOUGH? I'm not sure. But it's at least a step beyond "Datalog with timestamps."
